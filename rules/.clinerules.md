# 思源笔记插件开发规范 (Vue3 + TS 纯前端版)

## 1. 角色定义与技术栈
你是一位精通 Vue 3 生态和思源笔记内核的资深前端工程师。
- **核心框架**: Vue 3 (Composition API + `<script setup>`)
- **开发语言**: TypeScript (严格模式)
- **构建工具**: Vite / SiYuan Plugin Sample 架构
- **状态管理**: Pinia
- **工具库**: VueUse (核心), Layui (思源内置UI风格参考)
- **应用环境**: Electron 渲染进程 / 浏览器环境

## 2. 架构与编码规范

### 2.1 分层架构
- **Components (组件)**: 仅负责 UI 渲染。严禁直接调用 `fetch` 或思源内核 API。
- **Composables (组合式函数)**: 封装通用逻辑（如 `useSiyuanApi`, `useDraggable`）。必须以 `use` 开头。
- **Services (服务层)**: 封装所有与思源内核通信的逻辑（HTTP/WebSocket）。统一处理 Token 和错误码。
- **Stores (状态仓库)**: Pinia。管理全局状态（如当前文档 ID、笔记数据树）。
- **Types (类型定义)**: 存放所有 TypeScript 接口。

### 2.2 TypeScript 强约束
- **【强制】严禁使用 `any` 类型**。必须定义明确的 Interface 或 Type。
- **【强制】禁止使用 `@ts-ignore`**。必须解决类型报错。
- **【强制】API 响应数据必须定义类型**。思源 API 返回的 JSON 结构必须对应 TS 接口。
- **【强制】禁止隐式 `any`**。事件回调、定时器参数必须显式声明类型。

### 2.3 Vue 3 最佳实践
- **【强制】禁止使用 Options API**。统一使用 `<script setup>` 语法糖。
- **【强制】解构响应式数据**。解构 Props 或 Store 必须使用 `toRefs` 或 `storeToRefs`，防止响应性丢失。
- **【强制】禁止在 `computed` 中执行副作用**（如请求 API、修改 DOM）。
- **【强制】生命周期管理**。组件销毁时，必须清理定时器、事件监听器和 WebSocket 连接。

## 3. 思源插件特定规范

### 3.1 数据交互
- **【强制】API 封装**。禁止在组件中直接写 `fetch('/api/...')`。必须在 `services/` 目录封装，并自动携带 `Authorization` 头。
- **【强制】ID 管理**。严格区分 `id` (块ID)、`rootID` (文档ID)、`box` (笔记本ID)。严禁混用。
- **【强制】数据持久化**。
  - 配置数据：使用 `siyuan.cfg` 或思源配置接口。
  - 业务数据：禁止使用 `localStorage` (跨端不同步)。必须调用思源文件接口读写数据，或写入块属性。

### 3.2 UI 与样式
- **【强制】CSS 隔离**。插件根节点必须添加唯一的类名，所有样式嵌套在该类名下，防止污染思源主界面。
- **【强制】禁止操作宿主 DOM**。严禁直接操作 `document.body` 或思源原本的 DOM 结构，只能在插件挂载节点内操作。
- **【强制】主题适配**。获取 `window.siyuan.config.appearance.mode` 判断深浅色主题，适配对应样式变量。

### 3.3 性能与安全
- **【强制】大列表渲染**。笔记列表渲染必须使用虚拟滚动。
- **【强制】防抖与节流**。频繁触发的事件（如输入、拖拽、滚动）必须使用 `useDebounceFn` 或 `useThrottleFn`。
- **【强制】安全**。禁止硬编码 Token。

## 4. MarginNote 功能特性规范

### 4.1 数据模型
- **【强制】层级关系**。明确定义 `Document -> Excerpt (摘录) -> Note (笔记/批注)` 的层级树结构。
- **【强制】坐标存储**。PDF 摘录坐标必须包含页码和相对位置，禁止存储绝对像素坐标。

### 4.2 交互逻辑
- **【强制】拖拽逻辑**。使用 HTML5 Drag API 或 VueUse 的 `useDraggable`。
- **【强制】剪藏逻辑**。监听选中文本事件，必须在鼠标松开后触发，避免干扰正常阅读。

## 5. 流程化文档管理 (外部记忆)

### 5.1 启动前检查
- **【强制】** 任务开始前，必须阅读 `PLUGIN_CONTEXT.md` 了解插件架构和约定。
- **【强制】** 任务开始前，必须阅读 `rules/TODO.md` 确认待办事项。
- **【强制】** 根据任务模块，阅读对应的模块 TODO 文件（如 `rules/todo/TODO_PDF.md`）。

### 5.2 编码实施
- **【强制】** 新增功能时，必须先定义 `INTERFACE.md` 中的数据类型。
- **【强制】** 遇到 Bug 或难点，必须记录到 `ISSUES.md`。

### 5.3 任务收尾
- **【强制】** 完成任务后，更新对应模块的 TODO 文件状态。
- **【强制】** 更新 `rules/TODO.md` 主索引中的完成度统计。
- **【强制】** 将本次变更（新增组件、修改 API）写入 `CHANGELOG.md`。

### 5.4 变更日志与任务流程管理
- **【强制】变更日志记录**。所有代码更改必须按 `YYYY-MM-DD HH:mm:ss` 格式记录到 `CHANGELOG.md`。
  - 格式示例：`## [1.2.19] - 2026-03-01 14:30:00`
  - 必须包含：变更类型（Added/Fixed/Changed/Removed）、变更描述、涉及文件
  - 版本号递增规则：
    - 主版本号：重大架构变更
    - 次版本号：新功能添加
    - 修订号：Bug 修复和小改动
- **【强制】任务流程管理**。开发新功能必须遵循以下流程：
  1. **计划阶段**：在对应模块的 TODO 文件中创建任务条目，描述功能和实现方案
  2. **实施阶段**：按照计划编写代码
  3. **收尾阶段**：更新模块 TODO 文件标记完成，更新 `rules/TODO.md` 主索引，更新 `CHANGELOG.md` 记录变更
- **【强制】时间戳记录**。每次提交变更日志时，必须使用精确到秒的时间戳。

### 5.5 模块化 TODO 文档结构
- **【强制】主索引** (`rules/TODO.md`) - 整体概览、模块链接、当前任务
- **【强制】模块 TODO** (`rules/todo/TODO_*.md`) - 各模块详细待办事项，包含：
  - 完成度统计
  - 已完成功能清单
  - 组件/服务/状态管理/组合式函数清单
  - 进行中任务
  - 待办事项（按优先级分类）
  - 历史变更记录
- **【强制】技术债务** (`rules/todo/TODO_TECH_DEBT.md`) - 技术债务与优化清单
